# Privacy Policy — TronLink Extension

## 1. TL;DR — Our Promise in One Paragraph

TronLink Extension is a **non-custodial, client-side wallet**. Your private keys, mnemonic phrases,
wallet addresses, and transaction hashes **never leave your device** in plaintext. The only data we
transmit is **aggregated, anonymized usage statistics** that exist solely to help us improve the
product. We cannot link this data back to you, your wallet, your IP, or your browser — and the code
that proves it is open-source under the Apache-2.0 license.

If you only read one line of this document, read this:

> **We do not collect device-resident wallet addresses. We collect a random UUID that we generated
> for you locally, that you can wipe at any time by clearing extension storage, and that we cannot
> reverse into the address it represents.**

---

## 2. Design Principles

TronLink-Chrome is built around four invariants. Every code path that touches user data must satisfy
all four:

1. **Local-First** — Sensitive material (keys, mnemonics, addresses) is processed and stored
   on-device only. The backend is _unable_ to recover it, not merely _promising_ not to.
2. **Non-Identifying by Construction** — Telemetry payloads are designed so that no field, alone or
   in combination, can be tied to a real address, a real user, or a real transaction.
3. **Aggregated, Not Granular** — Raw values (token amounts, timestamps) are bucketed locally before
   any transmission. We see distributions, not data points.
4. **Auditable** — Every claim in this document maps to a specific file in the public source tree.
   If you do not trust the prose, read the code.

---

## 3. What Stays on Your Device, Always

The following items are generated, used, and stored **exclusively in your browser's local storage /
IndexedDB**, and are **never transmitted** to any server controlled by TronLink or any third party:

| Item                               | Where it lives                    | Sent to backend? |
| ---------------------------------- | --------------------------------- | :--------------: |
| Mnemonic phrase / seed             | Encrypted in extension storage    |    **Never**     |
| Private keys                       | Encrypted in extension storage    |    **Never**     |
| Public keys                        | Derived locally from private keys |    **Never**     |
| Wallet addresses (TRX / EVM)       | Extension storage + IndexedDB     |    **Never**     |
| Transaction hashes                 | Held in memory during signing     |    **Never**     |
| Contract call parameters / payload | Held in memory during signing     |    **Never**     |
| Counter-party addresses            | Held in memory during signing     |    **Never**     |
| Browser fingerprint / device ID    | Not collected                     |    **Never**     |
| IP address                         | Not stored or logged              |    **Never**     |
| Email / phone / KYC information    | Not requested                     |    **Never**     |

The wallet engine is **stateless on the server side**: there is no account to register, no email to
verify, no profile to build. TronLink-Chrome simply does not have these inputs to leak.

## 4. The Anonymous Analytics Module — How It Actually Works

To improve the extension, we need to answer questions like _"how many users staked TRX last week?"_
or _"what's the typical transaction-amount distribution on TRC-20?"_. We answer them **without ever
learning who any individual user is**. This section explains how.

The relevant code lives in
[`src/userStatistics/`](https://github.com/TronLink/tronlink-extension-core/tree/main/src/userStatistics).

### 4.1 Address → UUID: One-Way, Local-Only

When the analytics module needs an identifier for a wallet, it does **not** use the address.
Instead, it asks for or generates a random UUID:

```ts
// src/userStatistics/addressUuidMap.ts
import { ExternalDependencies } from './ExternalDeps';

export async function getOrCreateUuid(
  address: string,
  deps: ExternalDependencies,
): Promise<string> {
  return await deps.getOrCreateUuid(address);
}
```

The implementation behind `deps.getOrCreateUuid` does the following on first use of an address:

```ts
const uuid = crypto.randomUUID(); // fresh, cryptographically random UUID
await uidMappingStore.insert(address, uuid); // persisted on-device only (IndexedDB)
```

Why this matters:

- The mapping `address → uuid` exists **only inside your browser**. Our servers never see the
  address that produced the UUID.
- The UUID is generated by `crypto.randomUUID()` (RFC 4122 v4, 122 bits of entropy from the
  browser's CSPRNG). It is **not derived** from the address, public key, or any other property.
  There is no inverse function.
- Two wallets owned by the same person produce **two unrelated UUIDs**. The backend cannot cluster
  them into a single user.
- Wiping extension storage destroys the mapping. The next session generates a new UUID, breaking
  continuity with any prior reports.

### 4.2 Amounts → Logarithmic Buckets

Raw token amounts are never transmitted. Before a transaction record is queued, the amount is
replaced with a **9-bucket logarithmic histogram label**:

```ts
// src/userStatistics/userStatistics.ts
const ranges = [
  { range: '0_1', min: 0, max: 1 },
  { range: '1_10', min: 1, max: 10 },
  { range: '10_100', min: 10, max: 100 },
  { range: '100_1k', min: 100, max: 1_000 },
  { range: '1k_10k', min: 1_000, max: 10_000 },
  { range: '10k_100k', min: 10_000, max: 100_000 },
  { range: '100k_1m', min: 100_000, max: 1_000_000 },
  { range: '1m_10m', min: 1_000_000, max: 10_000_000 },
  { range: '10m_infinite', min: 10_000_000, max: Infinity },
];
```

A transfer of `1234.56 USDT` is recorded as bucket `1k_10k` with `count: 1`. The exact amount, the
recipient, and the transaction hash are discarded. The backend cannot reconstruct any specific
transaction from these histograms — it can only see, in aggregate, that "users made N transfers in
the 1k–10k range today."

### 4.3 Local Aggregation Before Upload

TronLink-Chrome merges records on-device, keyed by `(uid, actionType, tokenAddress, day)`,
**before** anything is sent. If you make ten TRC-20 transfers of similar magnitude on the same day,
the backend receives **one row** with `count: 10`, not ten individual events. Per-transaction timing
is destroyed by this step.

### 4.4 Mainnet Only

Telemetry is gated by `checkIsNeedReportChain()`. Activity on test networks (Shasta, Nile) and on
custom RPC endpoints is **not reported**. Developers and power users testing flows generate no
telemetry at all.

### 4.5 Encryption in Transit

Uploads go out via `checkAndReportXY(deps)`. Whether the payload is encrypted _on top of TLS_ is
**read from local config** each time, not hard-coded:

```ts
// src/userStatistics/report.ts
export async function checkAndReportXY(deps: ExternalDependencies): Promise<void> {
  if (!(await deps.checkIsNeedReportChain())) return;

  // …gather pending records, build reportXString / reportYString…

  const config = await deps.getSystemConfig();
  const { success, data } = await deps.sendStatistics({
    xMessage: reportXString,
    yMessage: reportYString,
    isEncrypt: isDef(config?.isEncryptReportData) ? config?.isEncryptReportData : true,
  });

  // The backend may flip the preference via `data.visible`;
  // the new value is persisted for subsequent uploads.
  if (config.isEncryptReportData != !data?.visible) {
    await deps.updateSystemConfig({
      ...config,
      isEncryptReportData: !data?.visible,
    });
  }
}
```

What this means concretely:

- **Default-on:** when no preference has been stored yet, payloads are encrypted
  (`isEncrypt: true`).
- **Config-driven:** the actual value is whatever the host's `getSystemConfig()` returns at call
  time.
- **Server-influenced:** the backend's response can flip the preference, and the new value is
  persisted via `updateSystemConfig` for future uploads. Disclosing this is intentional — it's an
  architectural fact, not a hidden one.
- **Transport:** TLS is provided by the host's `sendStatistics` implementation independently of the
  in-payload encryption flag.

### 4.6 Reported, Then Forgotten

Once a daily batch is acknowledged by the backend, it is **deleted from local storage** (today's row
is retained until UTC roll-over so subsequent same-day events can be merged). There is no rolling
local archive that could later be exfiltrated.

## 5. What the Backend Sees — Concrete Examples

To make the above tangible, here is the literal payload format used by the reporter:

**Asset snapshot (daily, per address):**

```
V1X|<uuid>|<addressType>|<trxBalance>|<usdtBalance>|<realTokenUsd>|<date>|
```

**Transaction record (daily, per token + action):**

```
V1Y|<uuid>|<addressType>|<actionType>|<count>|<tokenAddress>|<tokenAmountSum>|<energy>|<bandwidth>|<burn>|<date>|<distribution>|
```

Notice what is **not** in either payload:

- ❌ The wallet address
- ❌ The counter-party address
- ❌ The transaction hash
- ❌ Per-transaction amounts (only a daily sum `<tokenAmountSum>` and a 9-bucket histogram
  `<distribution>` are sent — see §4.2)
- ❌ Any timestamp finer than UTC date
- ❌ The IP, browser, OS, or extension version
- ❌ The user's chosen RPC endpoint

The `<tokenAddress>` field refers to the token contract being transferred (e.g., the USDT contract),
not your wallet. For TRX it is the literal placeholder `_`.

## 6. Why We Cannot Re-Identify You — Even If We Wanted To

A common, fair question: _"You promise not to link my UUID to my address — but couldn't you?"_

No. The architecture makes it impossible at the source:

1. **The address never leaves your browser.** The UUID-mapping table is written to your local
   IndexedDB. The backend has no API to read it. TronLink-Chrome has no code path that uploads it.
2. **The UUID is random, not derived.** There is no key, salt, or hash function that can map a UUID
   back to an address. The mapping exists only as the local IndexedDB row that you generated.
3. **Even with full server logs, we see no anchor.** No IP is associated with the report (the
   encrypted payload does not include one, and our infra strips per-user IP on ingest). No browser
   fingerprint, no device ID, no extension instance ID is sent.
4. **Multi-wallet users are indistinguishable from multi-user installs.** Because each address gets
   an independent UUID, a single person with 5 wallets is statistically identical to 5 different
   people with 1 wallet each.

The strongest privacy guarantees are the ones a vendor _could not violate_ even under coercion. This
is one of them.

## 7. What We Do **Not** Use Your Data For

We do not, and TronLink-Chrome does not enable us to:

- Sell or share usage data with advertisers, data brokers, or analytics vendors (Segment, Mixpanel,
  Amplitude, Google Analytics, etc.).
- Build a profile of your holdings, balances, or trading behavior.
- Cluster wallets by suspected ownership.
- Send you marketing emails (we do not have your email).
- Track you across websites, dApps, or sessions.
- Comply with subpoenas for individual user data — because we do not hold any.

## 8. Open Source — Verify, Don't Trust

Every claim above is testable. The full source is published at:

> **https://github.com/TronLink/tronlink-extension-core** — Apache-2.0

Recommended audit checklist:

- `src/userStatistics/addressUuidMap.ts` — confirm the address never leaves the local dependency.
- `src/userStatistics/userStatistics.ts` — confirm bucketing happens before any persistence.
- `src/userStatistics/report.ts` — confirm the on-wire format and that no address field exists.
- `src/userStatistics/constants.ts` — confirm chain-ID gating.
- `src/userStatistics/indexedDB/` — confirm mappings are local-only.
- `git log src/userStatistics/` — review the change history.

We welcome independent reviews and will treat reproducible privacy regressions as security
incidents.

## 9. Your Controls

- **Disable analytics entirely:** Toggle the analytics switch in the extension settings. When off,
  TronLink-Chrome skips both asset snapshots and transaction records (no payload is constructed or
  queued).
- **Wipe local UUIDs:** Clearing extension data destroys the `address → uuid` table. Future sessions
  will allocate fresh, unrelated UUIDs.
- **Use a non-mainnet network:** Activity on Shasta, Nile, or custom RPCs is never reported.
- **Inspect on-the-wire traffic:** All telemetry endpoints are documented in `report.ts`. You can
  confirm with browser DevTools that nothing is sent when analytics is disabled.
